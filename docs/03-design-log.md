# 03 · 설계 결정 기록 (Design Log)

**한국어** | [English](03-design-log.en.md)

이 프로젝트는 세 가지 작업 원칙 위에서 진행되었다.

1. **이론 먼저, 코드 나중** — 모든 결정은 구현 전에 이론적으로 정당화한다.
2. **한 축씩** — 여러 결정을 동시에 내리지 않고, 한 축을 대화로 확정한 뒤 다음으로 간다.
3. **절제** — 시네마틱한 과잉으로 미끄러지려 할 때마다 되돌린다. 덜 하는 것이 하는 것이다.

아래는 주요 결정의 시간순 기록이다.

## 2026-04-21 — 은유의 확립

ChatGPT 건설 안전 페르소나(위험 관리자 역할)와의 대화를 분석해 멀티에이전트 시스템의 구조를 추출하고,
이를 **도서관 여섯 개의 방**으로 매핑했다. 감리자 대시보드 와이어프레임(Figma, 1440×1040)과
인터랙티브 중간발표 HTML을 제작.

이 단계의 핵심 결정들:

- **비대칭 상호 가시성** — 작업자 간 비가시(데이터 순수성), 작업자→감리자 가시. 연구자 고유의 통찰이며 이후 모든 설계를 지배.
- **이중 언어 체계** — 페르소나가 지적한 "긴급 상황에서의 은유 해석 지연"에 대응.
  내부 서술은 도서관 언어, 화면 라벨·긴급 알림은 기능 언어. UI 토글로 전환.
- **"surface is message"** — 현장 가독성을 위해 상태를 텍스트 색이 아닌 배경 변화로 전달.
- **가장 본질적인 인터랙션의 선별** — 불일치 행의 붉은 맥박. 사용자의 주의를 요구하지 않고 작동하는,
  건축처럼 작동하는 유일한 인터랙션이라는 이유로 선택.
- **AI 페르소나를 설계 피어리뷰어로** — 설계안 제출 → 페르소나 비판 → 대응 → 재제출의 검증 사이클.
  네 가지 지적(은유 해석 지연 / "이상 없음" 습관화 / 감리자 화면 과부하 / 경고 피로)에 각각 대응.

## 2026-04-22 — 심사 피드백과 이론적 재정렬

*"가상공간 디자인인데 구조 설계에 치중되어 있다"* 는 심사 피드백.
이를 **System-as-space vs Designed space** 축으로 번역했다 — 공간 은유로 시스템을 서술하는 것과,
공간의 속성이 시스템의 변수가 되는 것의 구분.

"공간이 변수가 되어야 한다"는 네 방향(줌=관찰 거리, 체류=권한, 공간의 기억, 벽 두께=정보 속도)을
평가한 뒤, 압력에서 벗어나는 제3의 방향을 작업 가설로 잡았다 —
**가시성의 비대칭 자체를 설계 변수로 삼고, 방문자가 역할 사이를 이동하며 각 위치의 시야를 몸으로 알게 하는 공간.**
단, 이 질문은 확정하지 않고 미결로 유지하기로 결정했다.

## 2026-04-23 — 이론적 기반 정리

Mattern(Library as Infrastructure), Hillier(공간 구문), Fujimoto(친밀한 외부),
엔가와·마·오쿠 등 이론적 어휘를 정리하고 방별로 분배했다.
서양 이론과 일본 개념이 같은 것을 말하는 지점들(결절점≈오쿠, 친밀도 단계≈엔가와)을 확인 —
비대칭 가시성이 두 전통에서 동시에 토대를 가짐을 확인한 단계.

웹앱의 최종 형태 확정: **역할 선택형 SPA**, 단일 HTML, 5분 체험, 개발자 없이 Claude와 협업 구현.

## 2026-04 하순 — 작업자 화면 청사진

구현에 앞서 상세 청사진을 작성했다 (`worker_chamber_blueprint.md`). 핵심 결정:

- **매개의 5구간** — 입력(1초)·봉인된 흐름(2~3초)·진입(0.5~1초)·자기 응답(3~4초)·통제 명령(4~5초).
  각 구간이 자각·검토·개입 중 하나 이상의 조건을 만들어야 한다는 원칙.
- **유체 vs 결정** — 일상과 위험을 색이 아닌 *물질 상태*로 구분. 시각 언어의 토대.
- **분기 A/B** — 보고함/안 보고함에 따라 시스템의 처리와 체험자의 위치가 달라지는 구조. 논제의 체험적 증명.
- **안내창은 시스템 밖에** — 데모가 현장 감각의 결여를 보충하는 메타 도구이므로,
  박물관 캡션의 톤으로 시스템과 시각적으로 분리.
- **통제 명령은 개인이 아닌 환경에** — 명령의 결정 형태가 캡슐 안으로 흡수되지 않고 경계에 박혀 머무는 이유.

## 2026-05 ~ 06 — 구현과 통합

작업자·감리자 화면에 이어 관장실·서고·목록실을 구현하고 단일 파일로 통합했다.

- **공유 렌더링 엔진** — 의존성 없는 캔버스 Bayer 4×4 디더 엔진을 전 방이 공유.
  단일 빨강 `#E24B4A`는 위험·불일치에만 예약 (장식적 사용 금지).
- **관장실** — 위계 전체의 수직 척추. 세 결정(중지/속행/반려)의 빔이 서로 다른 범위에 도달 —
  권한의 범위를 빛의 도달 범위로 표현.
- **서고** — 기록↔장소 상호작용. 인덱스의 기록을 클릭하면 깊은 서가 속 위치가 점등.
- **목록실** — 조율 그래프 + 동기화된 스텝 로그 + 리플레이.
- **시스템 방의 진입 방식** — 목록실·서고는 역할 버튼이 아닌 도식의 노드 클릭으로 진입.
  사람의 자리가 아닌 시스템의 자리라는 구분을 진입 방식 자체로 표현.
- 통합 과정의 기술적 해법: 매 프레임 다시 그려지는 SVG 라벨과 클릭 대상을 분리
  (시각은 SVG, 클릭은 `getScreenCTM()`으로 매 프레임 정렬되는 고정 DOM 핫스팟).
